Βήμα 1 από 2 — Κύριος πίνακας δεδομένων:

-- ═══════════════════════════════════════════════════ -- GreenCut Pro v2 — Optimized Schema (v2.1) -- Εκτελέστε στο: Supabase Dashboard → SQL Editor -- ═══════════════════════════════════════════════════ -- [1/2] Κύριος πίνακας πελατών -- ─────────────────────────────────────────────────── CREATE EXTENSION IF NOT EXISTS "uuid-ossp"; CREATE TABLE IF NOT EXISTS public.clients ( -- UUID native type: πιο αποδοτικό από TEXT id UUID PRIMARY KEY DEFAULT gen_random_uuid(), created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), -- Στοιχεία πελάτη name TEXT NOT NULL, phone TEXT, address TEXT, -- Στοιχεία οικοπέδου sqm NUMERIC, price NUMERIC(10,2), -- 2 δεκαδικά για χρηματικά ποσά vegetation TEXT, work_date DATE, -- 'date' είναι δεσμευμένη λέξη SQL notes TEXT, -- NUMERIC για γεωγραφικές συντεταγμένες (όχι TEXT) -- Επιτρέπει μελλοντικούς χωρικούς υπολογισμούς lat NUMERIC(10,7), lng NUMERIC(10,7), -- Επαναλαμβανόμενες εργασίες frequency TEXT NOT NULL DEFAULT '0', next_date DATE, -- JSONB: αποδοτικότερο από JSON για αναζητήσεις tags JSONB NOT NULL DEFAULT '[]'::jsonb, -- Φωτογραφίες: Base64 (offline) ή Storage URL (cloud) -- Αν χρησιμοποιείτε Supabase Storage, εδώ αποθηκεύετε το URL photo_before TEXT, photo_after TEXT ); -- Ευρετήρια για απόδοση CREATE INDEX IF NOT EXISTS idx_clients_name ON public.clients(name); CREATE INDEX IF NOT EXISTS idx_clients_work_date ON public.clients(work_date DESC); CREATE INDEX IF NOT EXISTS idx_clients_tags ON public.clients USING GIN(tags); -- GIN για JSONB αναζητήσεις -- Trigger: αυτόματη ενημέρωση updated_at CREATE OR REPLACE FUNCTION update_updated_at() RETURNS TRIGGER LANGUAGE plpgsql AS $$ BEGIN NEW.updated_at = NOW(); RETURN NEW; END; $$; CREATE OR REPLACE TRIGGER clients_updated_at BEFORE UPDATE ON public.clients FOR EACH ROW EXECUTE FUNCTION update_updated_at(); -- ─── Row Level Security (RLS) ─────────────────────── ALTER TABLE public.clients ENABLE ROW LEVEL SECURITY; -- ΕΠΙΛΟΓΗ Α: Προσωπική χρήση (χωρίς Auth) -- Οποιοσδήποτε με το anon key έχει πλήρη πρόσβαση. -- Κρατήστε το anon key ΙΔΙΩΤΙΚΟ (μην το δημοσιεύετε). CREATE POLICY "anon_full_access" ON public.clients FOR ALL TO anon USING (true) WITH CHECK (true); -- ΕΠΙΛΟΓΗ Β: Πολλοί χρήστες (με Supabase Auth) -- Αντικαταστήστε την παραπάνω policy με αυτή αφού -- ενεργοποιήσετε Authentication στο Supabase Dashboard. -- CREATE POLICY "auth_user_access" -- ON public.clients FOR ALL TO authenticated -- USING (true) WITH CHECK (true);

Βήμα 2 από 2 — Supabase Storage (προαιρετικό, για φωτογραφίες cloud):

-- [2/2] Supabase Storage bucket για φωτογραφίες -- Εναλλακτικά αντί για Base64 στη βάση -- ─────────────────────────────────────────────────── -- Εκτελέστε αυτό μόνο αν θέλετε cloud αποθήκευση εικόνων. -- Μετά ενεργοποιήστε "Supabase Storage" στις Ρυθμίσεις. INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types) VALUES ( 'greencut-photos', 'greencut-photos', false, -- private bucket 3145728, -- 3MB max per file ARRAY['image/jpeg','image/png','image/webp'] ) ON CONFLICT (id) DO NOTHING; -- RLS για storage: μόνο anon/authenticated χρήστες CREATE POLICY "storage_anon_read" ON storage.objects FOR SELECT TO anon USING (bucket_id = 'greencut-photos'); CREATE POLICY "storage_anon_write" ON storage.objects FOR INSERT TO anon WITH CHECK (bucket_id = 'greencut-photos'); CREATE POLICY "storage_anon_delete" ON storage.objects FOR DELETE TO anon USING (bucket_id = 'greencut-photos');
⚠️ Σημείωση ασφαλείας (RLS):
Η Επιλογή Α (anon policy) είναι κατάλληλη για προσωπική χρήση — το anon key είναι ημι-δημόσιο, οπότε μην κοινοποιείτε το URL της εφαρμογής. Για πολλαπλούς χρήστες, ενεργοποιήστε Supabase Auth και χρησιμοποιήστε την Επιλογή Β.

🗂️ Αποθήκευση Φωτογραφιών

💾 Ανενεργό — Οι φωτογραφίες αποθηκεύονται ως Base64 απευθείας στη βάση.

💡 Πότε να ενεργοποιήσετε το Storage: Αν αναμένετε >50 πελάτες με φωτογραφίες, το Storage αποτρέπει «φούσκωμα» της βάσης και επιταχύνει τα queries. Απαιτεί εκτέλεση του Storage SQL (καρτέλα 🗄️).
